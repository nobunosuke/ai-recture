SYNC ／ PUSH

Push — 手元の変更を、
リモートに送り出す。

ローカルでいくらコミットしても、それはあなたの PC の中だけ。 他の人に届けるためにはリモートへ送り出す。それが Push だ。

1一言で言うと

Push とは、自分の手元にあるコミットを、リモート（GitHub）に送り出す動作。 ローカルで作った変更を、他の人に見える場所に「アップロード」する。

  • 方向は「ローカル → リモート」。送り出す動作。
  • push して初めて、他の人や PR の世界から見えるようになる。
  • コミット済みのものだけが送られる。未コミットの変更は対象外。

2なぜ必要なのか

Git の世界は、自分のローカルとリモートで別々に動いている。 ローカルで commit しても、それはあなたの PC の中だけの出来事だ。 他の人の画面には何も映らない。PR も作れない。

Push は、「自分の作業を共有可能な形にする」動作。 ここで初めて、レビューを依頼できるし、デプロイも走れる。 逆に言えば、push を忘れると「やったつもりで届いていない」事故が起きる。

push しないと

あなたの PC の中だけで完結する

他の人は何も知らない。PR レビューもデプロイも始まらない。「終わったよ」と言っても証拠が無い状態。

push すると

チームに共有された状態になる

GitHub の画面に反映され、他の人が見える。PR を立てられる。CI が走る。やっと「終わった」と言える。

3図で見る動き

ローカルに新しいコミット D・E を作った状態を想像してほしい。 リモートはまだ C で止まっている。git push を実行すると、D と E がリモートに送られて反映される。

「コミット」と「push」は別の動作

混同しやすいが、commit はローカルに保存、push はリモートに送信。 ローカルで何回 commit を重ねても、リモートには反映されない。最後に push して、はじめて共有される。

4実際の流れ

作業をひと区切りして、リモートに送るまでの典型的な流れ。

1

変更をコミットする

変更したファイルを git add → git commit -m "..." でコミット。ローカルに保存される。

2

念のため pull で最新を取り込む

push する直前に git pull。リモートが進んでいた場合に備える。

3

git push を実行

ローカルのコミットがリモートに送られる。初めて押すブランチなら -u origin <branch名> を付ける。

4

GitHub の画面で確認

リポジトリのページを開き、自分のコミットが反映されているか目で確認。これで PR の準備が整う。

5注意点

push は「送る」だけの単純な動作だが、リモートの状態次第で挙動が変わる。

うまくいくケース

リモートが自分の先頭の延長線上にある

「リモートはまだ C、自分は D まで進んだ」のようなパターン。素直に push が通る。

拒否されるケース

リモートが自分の知らないコミットを持っている

誰かが先に push していると、Git は「上書きしてしまうから止める」と拒否する。先に pull してから push する。

force push（--force）は危険

「拒否された push を強引に通す」オプションだが、他の人がそのブランチから始めた作業が消えることがある。 自分専用ブランチなら使う場面もあるが、共有ブランチでは絶対に使わない。

push したら「公開済み」

リモートに送ったコミットは、誰かが pull / clone した瞬間に手元に残る。 パスワードや API キーをうっかり push した場合、後から削除しても完全には消えない前提で、即座にローテーション（再発行）する。

6まとめ

push は「コミットしたものを、リモートに届ける」動作。これで初めて、変更がチームに共有される。

できること
手元のコミットを、リモート（GitHub）に送り出す。チームに共有する。
嬉しいこと
PR を出せる。CI が走る。レビューを依頼できる。「終わった」と言える。
使うタイミング
コミットがある程度たまった時、PR を出したい時、1 日の終わり。
注意点
拒否されたら先に pull。force push は使わない。秘密情報を含めない。
3 行で覚えるなら

① push は「コミットしたものをリモートに送る」動作。 ② push しないと他の人には届かない。 ③ 拒否されたら先に pull、force push は使わない。
