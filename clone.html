SYNC ／ CLONE

Clone — まるごと手元に、
最初の一回。

GitHub に置いてあるプロジェクトを、自分の PC で編集できるようにする「最初の一回」の動作。 ファイルだけじゃない。過去の履歴も、リモートとの繋がりも、まとめて持ってくる。

1一言で言うと

Clone とは、リモート（GitHub など）にあるリポジトリを、過去の履歴ごと丸ごと自分の PC にコピーしてくる動作。 一度クローンすれば、それ以降はずっと手元で作業できる。

  • 最初の一回だけの動作。2 回目以降は pull で更新する。
  • 履歴も全部入る。誰がいつ何を変えたか、すべて手元で追える。
  • リモートとの繋がりも自動で記録される。だからすぐに push / pull が使える。

2なぜ必要なのか

GitHub にプロジェクトがあっても、それはあくまでクラウド上の場所だ。 実際にコードを編集したり試したりするには、自分の PC に持ってこないといけない。 Clone はそのための「最初の一回」を担う動作だ。

「Web ブラウザで zip ダウンロードすればいいんじゃ?」と思うかもしれない。 でもそれだと、今の中身しか手に入らない。 Clone は違う。過去のすべての履歴と、リモートへの戻り道まで一緒に持ってくる。 だからその後の pull / push もすぐに動く。

zip ダウンロード

その時の中身だけが手に入る

過去の経緯も、リモートとの繋がりも無い。後で更新を受け取ったり、変更を送り返したりはできない。

git clone

履歴も繋がりも全部入る

過去のコミットも、リモートの場所も、全部記録された状態で手元に。すぐに pull / push できる。

3図で見る動き

左に GitHub（リモート）、右に自分の PC（ローカル）。 clone を実行すると、リモートの中身が丸ごと右側にコピーされて、新しいフォルダが手元に出来る。

「リモートへの参照」って何?

clone するとき、Git は「これは GitHub のこのアドレスから来た」という情報を一緒に記録する。 これが残っているから、後で pull / push するときに「どこと同期するか」を毎回指定しなくていい。

4実際の流れ

GitHub の画面から始まり、自分の PC にプロジェクトが落ちてくるまで。

1

GitHub でリポジトリの URL をコピー

リポジトリのトップで「Code」ボタン → URL（例: https://github.com/team/project.git）をコピー。

2

ターミナルで git clone を実行

git clone https://github.com/team/project.git のように、コピーした URL を続けて打つ。

3

同名のフォルダができる

カレントディレクトリに project/ という新しいフォルダが作られ、中にプロジェクト一式が入っている。

4

フォルダに入って作業開始

cd project でフォルダに入る。あとはブランチを切るも、編集するも、自由。

5注意点

clone は単純な動作だが、いくつか押さえておくと困らないポイントがある。

覚えておく

2 回目以降は pull

同じリポジトリをもう一度 clone する必要は無い。最新を受け取りたい時は pull を使う。

気をつける

同じ名前のフォルダがあるとエラー

すでに project/ が存在すると clone は失敗する。別の場所に移動するか、別名を指定して clone する。

プライベートリポジトリは認証が必要

非公開のリポジトリは、clone するときに GitHub アカウントの認証（SSH 鍵 or アクセストークン）が必要になる。 「URL は合ってるのに clone できない」時はまず認証を疑う。

6まとめ

clone は「リモートから、最初の一回、丸ごと持ってくる」動作。これさえ済めば、あとはずっとローカルで作業できる。

できること
リモートのリポジトリを、履歴と繋がりごと自分の PC にコピーする。
嬉しいこと
過去のコミットも、リモートへの戻り道も全部入る。すぐに pull / push できる。
使う頻度
1 つのリポジトリにつき、最初の 1 回だけ。それ以降は pull で更新。
注意点
同名フォルダがあると失敗する。プライベートは認証が必要。
