STEP 04 ／ PULL REQUEST

Pull Request — 取り込む前に、
見てもらう。

ブランチで作った変更を、いきなり main に流し込むのではなく、 「この変更を取り込んでください」と依頼する。 PR は、提案書とレビューの場を兼ねた、チーム開発の合流地点だ。

1一言で言うと

プルリクエスト（PR）とは、自分のブランチで作った変更を main に取り込むときに、 「取り込んでいいか、見てください」と依頼する提案書のことだ。 レビューを受けてから合流する、これがチーム開発の基本ルートになる。

  • 提案書として、何を・なぜ変えたかを残せる。
  • レビューの場として、他の人がコメント・指摘できる。
  • 取り込みボタンとして、承認後ワンクリックで main に統合できる。

2なぜ必要なのか

もし PR がなく、誰でも自分のブランチを直接 main に上書きできたとしたらどうなる？ 誰が、いつ、なぜ変えたかが分からない。間違いに気づける人もいない。 気付いたときには「動かないコードが本番に入っていた」という事故が起きる。

PR は、その手前に「説明と確認のフェーズ」を挟む仕組みだ。 ブランチで作業した本人が変更の意図を書き、レビュアーが内容を見て、議論する。 納得できたら統合する。記録と判断が同じ場所に残るところが本質だ。

PR を使わないと

本番に直接、無言で書き込む

変更の意図も経緯も残らない。間違いがあっても誰も止められず、後から原因を追えない。

PR を使うと

提案 → レビュー → 取り込み

変更の理由が文章で残り、第三者の目が入る。決裁書のように記録と判断がセットで残る。

名前は「Pull（取り込み）の Request（依頼）」

自分が書き込むのではなく、「本体側から自分のブランチを取り込んで（pull）ください」と 依頼する形式だから "Pull Request" と呼ぶ。主語が逆になっているのがポイント。

3図で見る動き

ブランチで積み上げたコミットを、レビューを通してから main に合流させる。 commit → PR → review → approve → merge の流れを横長で見ていく。

PR は「ブランチに紐づいた、議論の入れ物」

PR を開いた後にコミットを追加すると、その PR の中に自動で積まれていく。 別の場所に移動するわけではない。ブランチ＝作業、PR＝その作業に対する説明と議論と考えるとよい。

4実際の流れ

ブランチ summer-campaign で資料を更新した、という例で見てみよう。

1

ブランチで作業し、コミットを push する

編集して、変更を commit してから GitHub に push する。これで PR を開ける状態になる。

2

GitHub で「New Pull Request」を開く

取り込み先（main）と、取り込み元（summer-campaign）を選ぶ。差分が自動で表示される。

3

タイトルと本文を書く

「何を変えたか」「なぜ変えたか」「どう確認したか」を簡潔に書く。これがレビュアーの最初の手がかりになる。

4

レビュアーを指定して依頼する

誰に見てほしいかを指定。レビュアーは差分を見て、コメントや指摘を残してくれる。

5

指摘に対応して、追加コミットを積む

修正が必要なら、同じブランチで commit & push。PR は自動で追従するので、再依頼の操作はいらない。

6

Approve されたら「Merge」する

レビュアーが承認したら、Merge ボタンで main に取り込み。ブランチはそのまま削除して片付ける。

実際の PR は、こんな見た目をしている

GitHub 上の PR は、ヘッダー（状態とタイトル）と本文（説明）でできている。 本文はテンプレ通りに書けば、レビュアーが迷わない。

OPEN 夏キャンペーン用にヒーローのコピーを差し替える #42 ・ summer-campaign → main
何を（What）
トップページのヒーローキャッチを夏キャンペーン用の文言に差し替え、CTA ボタンの色を季節色に調整した。
なぜ（Why）
7/1 開始のキャンペーンに合わせて、トップを訪れたユーザーがすぐ気付ける見せ方にするため。マーケから依頼あり。
どう確認したか（How）
ローカルで表示確認 (Chrome / Safari)、スマホ幅でも崩れないことをチェック済み。

レビュアーとのやりとり

PR のコメント欄では、ブランチ上のコードに対して具体的な指摘ができる。 「ここ、こうしたほうがいいかも」「OK！」のやり取りがそのまま記録に残る。

R
reviewer レビュー 3分前
コピーいいですね。CTA の文字色がボタン背景と少し近くて読みにくいかも。 白に変えるとどうですか？
A
author 2分前
たしかに！白に変えました。コミット fix: CTA 文字色を白に 追加してあります。
R
reviewer Approved 1分前
バッチリ読めるようになりました。LGTM、マージしちゃってください。

5タイトルと本文の書き方

PR は後から読み返される。半年後に「なぜこの変更を入れたんだっけ？」と 振り返る相手は、自分かもしれないし、新しく入ったメンバーかもしれない。 書き方ひとつで、未来の調査時間が変わる。

タイトル：動詞 + 対象で、ひと目で分かるように

避けたいタイトル

修正 / 更新 / WIP

何のための変更か分からない。一覧で並んだとき、どれが何だったか区別がつかない。

分かりやすいタイトル

夏キャンペーン用にヒーローのコピーを差し替える

何を、どう変えるかが一行で伝わる。レビュアーは中身を開く前に文脈を掴める。

本文：3つの問いに答えるだけでいい

  • What — 何を変えたか: 変更の中身を 1〜3 行で。差分の要約だ。
  • Why — なぜ変えたか: 背景・依頼元・解決したい問題。いちばん重要。
  • How — どう確認したか: 動作チェックの方法。レビュアーが安心できる。
テンプレートを活用する

リポジトリに PR テンプレート（.github/PULL_REQUEST_TEMPLATE.md）を置いておくと、 PR を開いた瞬間に What / Why / How の見出しが入った状態で始まる。 毎回ゼロから書く負担が消える。

6まとめ

PR は「取り込んでください」と依頼する場所であり、 同時にレビュー・議論・記録が積まれていく場所でもある。 ブランチで手を動かし、PR で会話する。これがチーム開発のリズムだ。

できること
ブランチの変更を「提案」として開き、レビューを経て main に取り込める。
嬉しいこと
変更の意図と議論の経緯が記録として残る。後から「なぜこうした？」を辿れる。
注意点
レビューなしで自己マージしない。最低でも誰かに目を通してもらう習慣を作る。
書き方のコツ
タイトルは「動詞 + 対象」。本文は What / Why / How の 3 つを押さえる。
3 行で覚えるなら

① PR は「取り込んでください」とお願いする提案書。 ② 本文は What / Why / How を書けば 8 割 OK。 ③ Approve されてから Merge、そのままブランチは削除。
