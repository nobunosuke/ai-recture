分岐ルート ／ CONFLICT

Conflict — 壊れてない、
判断を待っているだけ。

Conflict は、エラーでも事故でもない。「同じ場所が 2 か所から書き換えられたから、どちらを残すか教えて」と Git が聞いてきている状態だ。 落ち着いてマーカーを読めば、数分で片付く日常作業になる。

1一言で言うと

Conflict（コンフリクト）とは、同じファイルの同じ場所を、別々の人が違う内容に書き換えた時に、Git が「どちらを残すか教えて」と聞いてくる状態のこと。 壊れたわけでも、間違ったわけでもない。Git が安全側に倒して、判断を人間に委ねている。

  • 異常じゃない。「判断保留」のお知らせ。出たら直すだけ。
  • 自動マージできない時だけ起きる。違う場所の編集なら Git が勝手に合流させてくれる。
  • 解決は「どっち残す？」を決めて書き直すだけ。難しい技術じゃなく、判断作業。

2なぜ起きるのか

ブランチを使えば、それぞれが独立して編集できる。だけど合流（マージ）するときに、 同じ行を二人とも別の内容に書き換えていたら、Git は「どちらが正しいか」を機械的に決められない。 「両方の言い分」をファイルに残して、人間に判断を委ねる — それが Conflict だ。

違う行を編集 自動マージ OK
1# 価格表（Aさん編集）
2通常価格
3新商品 A: ¥3,800（Bさん編集）
4在庫: 120 個

Git は両方を採用。ぶつかってないから判断不要。気付かないまま合流する。

同じ行を編集 ⚠ Conflict
1# 価格表
2通常価格
3新商品 A: ¥3,800 ← Aさん
3新商品 A: ¥4,200 ← Bさん

Git は判断保留。「3 行目、どっちが正解？」と聞いてくる。これが Conflict。

Conflict は「Git が壊れた」じゃない

Git は最後まで安全に振る舞っている。勝手にどちらかを上書きせず、両方の主張を残して人間に判断を求める。 これは「機能」であって、「不具合」じゃない。

3図で見る動き

左から右へ時間が流れる図。上の線が本番（main）、下の 2 本がそれぞれ price-aki と price-fuyu ブランチ。 両方が同じ price.md の 3 行目を別の値に書き換え、合流時に Conflict が起きる流れだ。

4マーカーの読み方

Conflict が起きると、ファイルの中に 3 本のマーカー線が自動で入る。 これは「Git が両方の主張をそのまま並べて見せている」状態。読めれば怖くない。

price.md
1# 価格表
2通常価格
3
4<<<<<<< HEAD # ← 自分の現在のブランチ
5新商品 A: ¥3,800
6======= # ← 境界線
7新商品 A: ¥4,200
8>>>>>>> price-fuyu # ← 取り込もうとしている向こうの版
9
10在庫: 120 個
<<<<<<< HEAD

ここから下が 自分の現在のブランチの版。HEAD は「いまチェックアウトしている方」を指す Git 用語。

=======

境界線。これより上が「自分の版」、下が「向こうの版」。

>>>>>>> price-fuyu

ここまでが 取り込もうとしているブランチの版。ブランチ名がそのまま入る。

つまり「上が自分、下が相手」を覚えれば 8 割。 あとは 3 本のマーカー線を消して、最終的に残したい版だけを書く。それで完了だ。

自分の版（HEAD）
新商品 A: ¥3,800

main を起点に price-aki ブランチが書き換えた値。

取り込む版（price-fuyu）
新商品 A: ¥4,200

合流させようとしている price-fuyu ブランチが書き換えた値。

最終版（自分で書く）
新商品 A: ¥4,200

営業に確認して新価格は ¥4,200 と判明 → こちらを採用してマーカーを削除。

5解決の流れ

実際の手順は 4 ステップ。難しい技術じゃなく、ファイルを開いて編集するだけだ。

1

Conflict したファイルを開く

Git が「ここで止まったよ」とファイル名を教えてくれる。エディタで開けば、マーカーが入った 3 本線が見つかる。

2

どちらを残すか判断する

マーカーの「上が自分・下が相手」を見て、最終的な正解を決める。両方混ぜることも、片方を捨てることもできる。迷ったら作業した人に聞く。

3

マーカー 3 行を消して、最終版だけ残す

<<<<<<< と ======= と >>>>>>> の 3 行は必ず削除。残すのは「自分で書いた最終的な版」だけ。

4

保存して「解決した」とコミットする

保存して、変更をコミット。これで Git は「合流できた」と認識する。Conflict は終了。

マーカーの消し忘れに注意

<<<<<<< や >>>>>>> を消し忘れるとファイルが壊れる。 「3 本のマーカー線を全部消したか？」をコミット前にもう一度確認すること。

6怖くないコツ

Conflict は「起きる前にどう減らすか」と「起きた時にどう向き合うか」の 2 軸で考えると気持ちが楽になる。

気持ちのコツ

  • Conflict は壊れた状態じゃない。Git が判断を待ってるだけ。
  • 判断材料は全部ファイルに揃ってる。自分の版と相手の版が両方見えてる。
  • 分からなければ書いた人に聞く。これは技術じゃなく、合意の作業。

予防のコツ

  • こまめに合流する。差分が小さいほど衝突しにくい。週単位より日単位。
  • 同じ場所をいじりそうな時は声をかける。「いま価格表触ってる」の一言で防げる。
  • 作業前に main の最新を取り込む。古い起点から始めるほど、合流時に衝突しやすい。
Conflict が増えるチーム

長期間ブランチを抱える

1 ヶ月誰にも言わずに作業 → 久しぶりに合流したら同じ場所が 5 か所もぶつかってた、というやつ。

Conflict が減るチーム

毎日合流、毎日声かけ

こまめに main を取り込み、被りそうな時は事前に共有。起きても 1〜2 行で済む。

7まとめ

Conflict は「同じ場所が 2 か所から書き換えられた」時に Git が判断を委ねる仕組み。 読み方さえ覚えれば、難しい技術じゃなく日常の判断作業に変わる。

何が起きてる
同じファイルの同じ行が、2 か所から別々に編集された。
なぜ起きる
Git が安全側に倒して、勝手に上書きせず人間に判断を委ねる。
解決の鍵
マーカーは「上が自分、下が相手」。最終版を書いてマーカー 3 行を消す。
予防策
こまめに合流、被りそうな場所は声かけ、作業前に最新を取り込む。
3 行で覚えるなら

① Conflict は「どっち残す？」と聞かれてる状態。壊れてない。 ② マーカーの上が自分、下が相手。最終版を書いて 3 本のマーカー線を消す。 ③ こまめに合流しておくと、そもそも起きにくい。
