SYNC ／ PULL

Pull — 最新を、
手元に取り込む。

自分が作業している間にも、リモートでは他の人がどんどんコミットしている。 古い状態の上で作業しないために、まずリモートの最新を取り込む。それが Pull。

1一言で言うと

Pull とは、リモートの最新状態を自分の手元に取り込む動作。 他の人の変更を受け取って、自分のローカルを最新に追いつかせる。

  • 方向は「リモート → ローカル」。受け取る動作。
  • 作業を始める前と、共有する前 に必ずやる。
  • 自分のローカルが新しい状態に追いつく。逆に自分の変更が消えるわけではない。

2なぜ必要なのか

あなたが昨日 push して帰った後、夜中に同僚が新機能を main にマージした。 翌朝、あなたが何もせずに作業を始めたら、「昨日の状態」の上に積み上げてしまう。 これだと、後で push しようとしたときに「あなたのローカルは古い」と怒られる。

Pull は、この「気付かないうちに古くなる」問題を防ぐ動作だ。 リモートとの差分を取り込んで、ローカルを最新の状態に揃える。これで安心して作業に入れる。

pull せずに作業すると

古い土台の上で進めてしまう

他の人が直したバグを、自分が知らずに別の形で直す。push する時に衝突して、後から面倒な調整になる。

pull してから作業すると

最新の土台の上で始められる

他の人の変更を踏まえて作業できる。重複や無駄な手戻りが起きにくい。

3図で見る動き

リモートに新しいコミット D が追加された状態を想像してほしい。 ローカルはまだそれを知らない。git pull を実行すると、D が手元に降りてくる。

pull は「fetch + merge」をまとめてやってる

正確には、リモートから情報を取得（fetch）して、それを自分のブランチに統合（merge）する 2 ステップを一発でやってくれるのが pull。 最初は深く考えなくて良いが、「2 つの動作をまとめたもの」とだけ覚えておくと、後で疑問が解ける。

4実際の流れ

朝、作業を始めるときの典型的な流れ。

1

作業フォルダに入る

ターミナルで cd project。clone 済みのフォルダに移動する。

2

git pull を実行

リモートと自分のブランチを比べて、足りない分だけ持ってくる。何も無ければ Already up to date. と出る。

3

差分が表示される

「○○ファイルが変更されました」とリストが出る。これで何が更新されたか分かる。

4

そのまま作業開始

ローカルが最新に揃ったので、安心してブランチを切ったり、編集したりできる。

5注意点

pull は基本的に「降りてくるだけ」だが、状況によっては衝突（Conflict）が起きることもある。

うまくいくケース

自分のローカルが綺麗な状態

未コミットの変更が無いか、あっても別のファイルなら、pull はすんなり通る。普段はこちらが多数。

つまずくケース

同じ箇所を双方が編集していた

リモートと自分の両方が同じファイルの同じ行を変えていると、Conflict（衝突）が起きる。手で調整が必要。

pull の前に「未保存」を片付けておく

作業途中で pull したいときは、未コミットの変更を先にコミット or 退避（stash）しておくのが安全。 そのまま pull しようとすると「上書きされる可能性があるよ」と Git に止められることがある。

pull するタイミングは「作業開始前」と「PR 出す前」

この 2 つを習慣にしておけば、ほとんどの衝突は防げる。朝来たら pull、push する直前にも pull。これだけ。

6まとめ

pull は「リモートの最新を、手元に揃える」動作。やる時は意識して、サボらない。

できること
リモートの新しいコミットを、自分のローカルに取り込む。
嬉しいこと
他の人の変更を踏まえて作業できる。push で衝突するリスクが減る。
使うタイミング
作業を始める前、PR を出す前。1 日に複数回やる。
注意点
未コミットの変更があると止められることがある。先にコミット or stash。
3 行で覚えるなら

① pull は「リモートの最新を取り込む」動作。 ② 作業前と push 直前にやる。 ③ 未保存変更があるなら、先にコミット or stash。
