ホーム / Issue（イシュー）
STEP 01 ／ ISSUE

Issue — やることを、
1 件ずつ切り出す。

「ちょっとあれやっといて」は、口にした瞬間に消えていく。 イシューは、その依頼を 1 枚のカードとして残す仕組みだ。 誰が・何を・いつまでに・どうなったら終わりか。形にして見える場所に置く。

1一言で言うと

Issue とは、「やること 1 件分のカード」のことだ。 プロジェクトでやるべき作業を 1 つずつ切り出して、見える場所に貼り出すための仕組み。

  • 1 件 = 1 タスク。小さく切って、独立して扱える単位にする。
  • タイトルと本文で、何をやるか・なぜやるか・どうなったら終わりかを書く。
  • 終わったら閉じる（Close）。やったことが台帳として残り、後から辿れる。

2なぜ必要なのか

チャットで「あれ、お願いね」と頼まれた経験はあるはずだ。数日後、相手はもう忘れている。 自分も「やった気はするんだけど、ちゃんと終わったんだっけ？」と曖昧になる。 口頭やチャットの依頼は、その場で流れていってしまうのが普通だ。

Issue は、この問題を構造で解決する。依頼を 1 枚のカードにしてしまえば、 「誰の宿題か」「いつ作られたか」「終わったか」が見える場所に残る。 チャットの会話と違って、Issue は閉じるまで消えない。

Issue を使わないと

チャットで「あれやっといて」

会話の中に依頼が埋もれる。誰宛か曖昧、完了したかも分からない。1 週間後には全員が忘れている。

Issue を使うと

1 件のカードとして起票する

担当・期日・完了条件が 1 か所にまとまる。一覧で「残っている宿題」が見える。終わったら Close。

3Issue の中身を見る

実際の Issue を 1 枚、拡大して中身を見てみよう。下の図は GitHub の Issue を簡略化したもの。 本文（左）と メタ情報（右） の 2 つのエリアに分かれている。

「Open」「Closed」は Issue の唯一の状態

Issue は基本的に Open（やる予定 / やってる）か Closed（終わった）の 2 つしかない。 「いま誰の手元にあるか」「どこまで進んでいるか」はメタ情報（Assignee や Linked PR）と本文のコメントで補う。

4実際の流れ

起票から Close までを、1 つのキャンペーン作業を例に追ってみる。Issue は 「最初に立てて、最後に閉じる」のが基本だ。

1

Issue を起票する

「夏のキャンペーン用 LP を作る」というタイトルを付けて、背景・やること・完了条件を書く。これで依頼が形になる。

2

担当（Assignee）を決める

自分でやるなら自分にアサイン。チームで分担するなら担当者を指定。これで「誰の宿題か」が見える化される。

3

その Issue を解決する Branch を切る

Issue 番号（例: #12）を含むブランチ名にすると、後で「どの Issue のための作業か」を辿りやすい。例: 12-summer-campaign-lp。

4

作業して PR を出す

PR の本文に Closes #12 と書いておくと、PR がマージされた瞬間に Issue が自動で Close される。手で閉じる手間が省ける。

5

Close（自動 or 手動）

完了。閉じられた Issue は消えるわけではなく、台帳に残る。「あの作業っていつ・誰が・なぜやったんだっけ？」を後から辿れる。

5良い Issue の書き方

半年後の自分や、ジョインしたばかりのメンバーが見て「何の話か一目で分かる」のが良い Issue だ。 タイトルと本文の書き方で、Issue の質はガラッと変わる。

避けたい書き方

「サイト直す」（本文なし）

どこを、なぜ、何が直ったら完了なのかが分からない。1 週間後の自分でも内容を思い出せない。

伝わる書き方

「ヘッダーの背景色を白に戻す」

本文に「背景: 先週のリリースで誤って黒に変わった / 完了条件: 全ページのヘッダー背景が #FFFFFF に戻っている」。半年後でも辿れる。

書くときに意識する 3 つは以下の通り。最低限これだけ揃えば、十分に「動く Issue」になる。

  • タイトルは「動詞 + 対象」。修正 や 更新 のような曖昧な動詞は避け、何をするかを具体的に。
  • 本文は「背景 / やること / 完了条件」の 3 段。なぜ必要かを 1 行、やることを箇条書きで、終わりの定義を 1 行。
  • ラベルで種類を示す。bug、feature、docs などで色分けすると、一覧で性質がひと目で見える。

6まとめ

Issue は「やることを 1 件のカードにして、見える場所に置く」仕組み。 流れて消える依頼を形にして残すのが本質だ。

できること
依頼を 1 件のカードにして、誰が・何を・いつまでに・どうなったら終わりかを残す。
嬉しいこと
チャットで流れて消えない。状態が見える。終わったら台帳として残る。
注意点
タイトルだけだと薄い。本文に「背景」と「完了条件」を 1 行でいいから書く。
名前のコツ
「動詞 + 対象」で、半年後の自分が見て中身を想像できる粒度に。
3 行で覚えるなら

① Issue は「やること 1 件分のカード」。 ② 流れて消える依頼を、形にして見える場所に残す。 ③ タイトルは「動詞 + 対象」、本文に「背景」と「完了条件」を 1 行。
