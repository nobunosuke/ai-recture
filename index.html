Step by step

作業の流れを、4 つの言葉で覚える。

Git の作業は順序がある。Issue で「やること」を切り出し、Branch で作業場所を分ける。 Worktree で実際に編集し、PR で取り込みを依頼する。途中で Conflict が起きることもある。

準備中 1STEP 01

Conflict

同じ場所を別々に編集してしまった時に起きる、どちらを残すかの調整作業。

詳しく見る

Conflict は通常の流れの「外側」で起きる例外。 複数人が同時に同じファイルの同じ箇所を編集すると、Git は自動的に判断できず、人間に判断を委ねる。

Local ⇄ Remote

もう 1 つの軸：
ローカルと、リモート。

ここまでは「自分の手元」の流れ。Git にはもう一つ、自分の PC（ローカル）と共有サーバー（リモート、たとえば GitHub）の間でやり取りする仕組みがある。3 つの動作だけで覚えられる。

詳しく見る UPLOAD

Why we learn

なぜ Git を学ぶのか。

Git は「ファイルの歴史を全部覚えてくれる仕組み」だ。やったことを取り消せる安心と、複数人で同じ作業に手をつけても壊れない安全がある。

01 / 並行

同時に動ける

同じ資料を別々の人が編集しても、上書き合戦にならない。それぞれの作業を後から繋ぎ合わせられる。

02 / 復元

いつでも戻せる

「昨日の状態に戻したい」が一行で叶う。失敗しても怖くないから、思い切って試せる。

03 / 共有

変更の理由が残る

いつ・誰が・なぜ変えたかが記録される。半年後の自分やチームが過去を辿れる。
