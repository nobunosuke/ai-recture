エンジニアが毎日 Git で何をやっているのか。バラバラの用語を覚える前に、まず作業全体の流れをつかむ。 この資料は、5 つの基本概念を順番に追いながら、それぞれのつながりを地図のように理解するための入門書だ。

Map of the words

5つの言葉を、流れで覚える。

Git の作業は順序がある。Issue で「やること」を切り出し、Branch で作業場所を分ける。 Worktree で実際に編集し、PR で取り込みを依頼する。途中で Conflict が起きることもある。

1STEP 01

Issue

やることを 1 件ずつ切り出して、見える場所に置く。出発点。

詳しく見る 2STEP 02

Branch

いまの状態を壊さずに、別バージョンで作業するための分身。

詳しく見る 3STEP 03

Worktree

ブランチごとに作業フォルダを分けて、並行で進められる仕組み。

詳しく見る 4STEP 04

Pull Request

「この変更を取り込んでください」と依頼する、レビュー付きの提案書。

詳しく見る
準備中 !分岐ルート

Conflict

同じ場所を別々に編集してしまった時に起きる、どちらを残すかの調整作業。

詳しく見る

Conflict は通常の流れの「外側」で起きる例外。 複数人が同時に同じファイルの同じ箇所を編集すると、Git は自動的に判断できず、人間に判断を委ねる。

Why we learn

なぜ Git を学ぶのか。

Git は「ファイルの歴史を全部覚えてくれる仕組み」だ。やったことを取り消せる安心と、複数人で同じ作業に手をつけても壊れない安全がある。

01 / 並行

同時に動ける

同じ資料を別々の人が編集しても、上書き合戦にならない。それぞれの作業を後から繋ぎ合わせられる。

02 / 復元

いつでも戻せる

「昨日の状態に戻したい」が一行で叶う。失敗しても怖くないから、思い切って試せる。

03 / 共有

変更の理由が残る

いつ・誰が・なぜ変えたかが記録される。半年後の自分やチームが過去を辿れる。
